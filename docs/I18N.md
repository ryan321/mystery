# Internationalization (i18n) Analysis

Status: analysis only — no code changes. Scope: offering the site (UX),
gameplay, and case content in multiple languages.

Current state in one sentence: **every layer is hardcoded English** — there
is no i18n framework, no locale field anywhere in the data model, and
several correctness-critical code paths parse English text with regexes.

---

## 0. The decision that shapes everything

Before any extraction work, one design question has to be answered:

**A. Translate the case definitions** — each case ships per-language
(`definition.json` translated, ~78k chars of prose per case × 9 cases).
The LLM then narrates naturally in-language because its source material is
in-language.

**B. Keep cases English; the LLM performs in the target language** — inject
"respond in language X" into the performer prompt. Zero content pipeline,
works the day it ships, but names/clues/fair-play wording are
LLM-translated on the fly with no editorial control, and quality varies by
language and model.

`docs/STEAM.md:174` already sketches the intended answer: *"Localization
later: store page + UI first… performed-dialogue localization only when a
market proves out."* That maps to a **phased approach: B first (UI + prompt
locale), A later per-language for proven markets.** This doc assumes that
phasing but flags where A would change the work.

The hybrid risk to manage under B: engine *regexes and scorers* that read
English text (§3) will be reading non-English player input and non-English
LLM output. Those must be fixed regardless of A or B.

### Decision log

- **2026-07 — Accusation scoring: "extract, don't match" (implemented).**
  The verdict is never an open LLM judgment (non-deterministic,
  prompt-injectable, unauditable). Instead an LLM *extraction* call
  reports WHAT the accusation claims — `namedCulpritIds` + per-fact
  `affirmed | denied | unmentioned` (`AccusationExtractionSchema`,
  `packages/shared/src/pipeline.ts`; call site
  `packages/llm/src/extract-accusation.ts`) — and the engine decides
  win/lose deterministically from that structure
  (`scoreAccusationDetailed`'s optional `extraction` param). Negation and
  paraphrase are handled semantically, so this works in any language and
  also fixes English paraphrase edge cases. Extraction failure falls back
  to the legacy negation-aware regex matcher, which remains the
  mock/offline path. Being validated on English first before any
  non-English gameplay ships. The same pattern is the model for the
  remaining traps below (boundary detection → closed-enum director field;
  presence validator → structured output or downgrade to warning).

## 1. Web UX (`apps/web`)

### 1.1 String extraction (~300–400 strings, ~35 files)
No i18n library today (no `next-intl`/`react-i18next`; no `i18n` block in
`next.config.ts`). Strings live inline in JSX across 14 pages + 21
components + `lib/format.ts` label maps + `lib/ambience.ts` track/pack
names. Kinds: headings, marketing prose (landing `app/page.tsx` is 485
lines of it), button labels, placeholders, aria-labels/titles, error
messages, enum→label maps (`difficultyLabel`, `willingnessLabel`,
`lockLabel`).

Recommended: `next-intl` (App-Router-native), one message catalog per
locale, ICU MessageFormat for plurals/interpolation.

**Traps found:**
- Manual pluralization that must become ICU plurals:
  `lib/format.ts:152` (`turn${turns === 1 ? "" : "s"}`),
  `components/DenouementBanner.tsx:22-26`, `components/Log.tsx:236`
  ("Load earlier (N more)"), `lib/format.ts:21` ("Solve N more…").
- Mid-sentence interpolation that breaks word order:
  `CastList.tsx:87` / `PresenceStrip.tsx:32` (`View ${knownAs}`),
  `gallery/page.tsx:227` (`Remove ${t}`).
- **Not all strings are UI copy:** `app/play/[id]/page.tsx:455-456`
  (`Return to the ${loc.name}.` / `Head to the ${loc.name}.`) is *sent to
  the game engine as player input* — it must be built in the input
  language, not extracted for display translation. Extraction passes must
  whitelist, not blanket-sweep.
- `lib/format.ts:40` `DIFFICULTY_TAG_RE` filters English tag words out of
  theme chips — assumes English content tags.

### 1.2 Routing / locale segment
No `[lang]` segment, no middleware. Adding one touches:
- `app/layout.tsx:28` `metadataBase` hardcoded; `<html lang="en">`
  hardcoded at `:58`; root layout moves under `[lang]`.
- `app/sitemap.ts` (per-locale `alternates.languages`), `app/robots.ts`,
  `app/manifest.ts` (`name`/`description`/`start_url` — manifest can't
  live under `[lang]` without extra work).
- `next.config.ts:7-12` `/shelf → /gallery` redirect is locale-unaware.
- Auth flow path plumbing: `safeNext` in `signin/page.tsx:11-14` and
  `signup/page.tsx:11-14`, magic-link verify redirect
  (`signin/verify/page.tsx:20-23`), `MysteryDetail.tsx:89,96`
  `router.push('/signup?next=…')`, `GoogleButton.tsx:19` — all need
  locale-prefixed paths.
- OG image generator `app/mystery/[id]/opengraph-image.tsx` renders
  English text via `next/og`; satori needs non-Latin fonts for CJK etc.

### 1.3 Fonts
`layout.tsx:7-21` loads Cinzel / Source Sans 3 / IBM Plex Mono with
`subsets: ["latin"]` only.
- Source Sans 3 + IBM Plex Mono: cyrillic/greek/vietnamese/latin-ext
  subsets exist — just add them.
- **Cinzel ships latin + latin-ext only** — needs a fallback/companion
  display face for Cyrillic/Greek.
- **CJK: none of the three families cover it** — new font stack + CSS
  variable story (`--font-cinzel` etc. are applied on `<html>`).

### 1.4 Formatting
Green field: zero `Intl.*`/`toLocaleString` usage today. Numbers render
raw; dates (`updatedAt`, `freeUntil`, note timestamps) are stored but never
displayed yet. Establish `Intl.DateTimeFormat`/`NumberFormat` helpers now
so future UI is locale-correct from day one.

### 1.5 Storage
All localStorage/sessionStorage keys hold enums/JSON — locale-independent,
no migration needed. One wart: `sessionStorage` caches the English
`opening`/`briefing` per playthrough (`MysteryDetail.tsx:21-30`,
`play/[id]/page.tsx:314-323`) — a mid-session locale switch replays stale
English. Key those by locale once locales are switchable. A new
`mystery.locale` pref key joins the existing set (note the codebase mixes
`mystery.` and `mystery:` separators already).

### 1.6 API-supplied strings rendered raw
- `lib/api.ts:65` throws server `message` prose that the UI displays
  verbatim (`MysteryDetail.tsx:94-99`, `play/[id]/page.tsx:441`,
  `NotebookPanel.tsx:39`). Only `signin_required` is handled as a machine
  code. **The API must move to error codes; the web maps codes → localized
  messages.**
- Engine-generated progress labels (`MysteryProgress.depthLabel` /
  `throughLabel` — "About two-thirds through",
  `packages/engine/src/progress.ts:52-80`) ship to the client. Either make
  the engine locale-aware or ship structured data and format client-side
  (`throughCompact` "≈⅔" is already locale-safe).
- `gallery/page.tsx:33-48` hardcodes a full English fallback case when the
  API is down.

## 2. Gameplay / LLM layer (`packages/llm`, `packages/engine`, `apps/api`)

### 2.1 Prompt surface
Four LLM call sites, each with its own hardcoded-English system prompt:

| Call site | Prompt | Produces |
|---|---|---|
| Director (intent parser) | `packages/llm/src/director.ts:25-72` | structured intents (JSON) |
| Performer (narrator) | `packages/llm/src/performer.ts:21-67` | **all player-facing prose** |
| Physical classifier | `packages/llm/src/classify-physical.ts:30-57` | structured JSON |
| Legacy narrator | `packages/llm/src/prompts.ts:3-31` | (off the main pipeline) |

English also enters via: `STATIC_POLICY` (10 policy strings,
`packages/engine/src/static-pack.ts:22-50`), the static case block and
context pack (authored case prose — `static-pack.ts:56-95`,
`context-pack.ts`), engine-built stage directions fed to the performer
(`apps/api/src/turn-pipeline.ts:105-108` and ~6 more sites,
`boundaries.ts:160-196`, `accuse-gate.ts:147-155`,
`accusation.ts:392-401`), and English retry/repair messages
(`client.ts:215,440,456`).

### 2.2 Locale plumbing (the easy part)
There is **no locale field anywhere** — add one to `PlaythroughState` /
creation (`create-playthrough.ts`, `apps/api/src/index.ts` start route),
sourced from the user's preference (see §4) with `Accept-Language` as a
hint. Inject "respond in X" into **three** prompts (performer, director,
classifier) — there is no single choke point. Prefix-cache constraint: the
static case block is memoized byte-identical per definition
(`static-pack.ts:97-109`), so any per-language content or instruction in
that block must key the cache by language too (and splits provider
prefix-cache hits per language).

### 2.3 English-dependent parsing — the real risk, ordered by danger

1. **Accusation scoring decides win/lose** —
   `packages/engine/src/accusation.ts:52-75` matches the player's free-text
   accusation against authored English `matchHints` with English negation
   regexes (`NEG_BEFORE_RE`/`NEG_AFTER_RE`: "not", "n't", "doubt",
   "innocent", "exonerat*"). *"Vale war es nicht" scores as an affirmative
   accusation.* Silent false convictions/acquittals. Fix options: LLM-based
   scoring against authored facts, per-language hint lists in the bundle,
   or per-language negation grammars. **This is the single most dangerous
   spot in the codebase for i18n.**
2. **Narration presence validator** — `performer.ts:134-203` regex-matches
   generated narration against English presence verbs ("stands|sits|
   watches…") and prepositions. Breaks on translated narration; the
   soft-retry loop would misfire or never fire. Replace with structured
   performer output or per-language validators.
3. **Player-input classifiers (English regex packs)** —
   `boundaries.ts:22-129` (jailbreak/abuse/impossible — *the local safety
   net*), `accuse-gate.ts:16-25` (a pending accusation can never be
   confirmed in another language), `intents-to-patch.ts:134-186`
   (assault/misconduct verb lists), `turn-pipeline.ts:311-317` (goodbye
   regex), plus both heuristic fallbacks (`heuristic-director.ts`,
   `heuristic.ts` — extensive keyword logic incl. hardcoded Blackwood
   names). Options: per-language regex packs, or move detection fully to
   the LLM director (which already duplicates boundary detection).
4. **Machine tokens must stay English** — director/classifier must
   *understand* language X but keep emitting English `physical.kind`
   (matched at `classify-physical.ts:217-230`, `director.ts:113-124`),
   boundary notes (matched at `boundaries.ts:86-129`), `exit_denouement`,
   dressing `subject` slugs. Enums are already Zod-locked (good); the free
   strings need explicit pinning instructions and ideally schema
   tightening.

### 2.4 Voice / period flavor
Character voice is authored case content (`characters[].voice`,
`player.voiceNotes`, `wrapUp.performanceNotes`) — e.g. london-1888: "Blunt,
procedural, East End vowels", "Sharp, Cockney-tinged educated", "Victorian
formality". These are **English-specific concepts with no direct mapping**
into most target languages. Under phase B the LLM improvises; under phase A
a localization designer must re-author voice per language. This is a
creative problem, not a string swap — flag it to authors early.

### 2.5 Token budgets / storage
- `DEFAULT_MAX_OUTPUT_TOKENS = 3000` (`client.ts:54`); CJK/Devanagari
  tokenize 2–4× denser — narration headroom tightens; recheck budgets per
  script family.
- Under phase A, each language ships a full copy of every case — bundle
  storage multiplies (~139 KB JSON × languages × cases), static-pack cache
  keys and provider prefix caches split per language.

## 3. Case content & publishing (`content/`, `apps/api/src/registry.ts`, `apps/studio`)

### 3.1 Translatable-field manifest
`definition.json` (~139 KB, 22 top-level sections, ~78k chars of prose for
blackwood) mixes prose with identifiers. The schema has no translatable
marker; a field-level manifest must be authored (section-by-section map
already derived in this analysis — characters ~24k, locations ~14k,
canon ~12k, beats ~5k, revelation ~4k, endings ~4k, evidence ~3.5k,
solution ~3k, player/briefing ~2.7k). Distinctions that matter:
- **Player-visible verbatim**: `meta.title/premise`, `openingNarration`,
  `revelation`, evidence names/descriptions, location names, cast card
  fields, ending titles, briefing.
- **AI-context only**: canon, secrets, beats' `narrationHints`, voice notes
  — still needs translation under phase A (prompts read them).
- **Dual-use logic+prose**: `solution.rubric.requiredFacts[].matchHints`
  feed accusation matching (§2.3.1) — translations must stay aligned with
  the accusation language. `relationships[].type` is a required free
  string that reaches prompts.
- `lintBundle`'s name-leak checks (`apps/api/src/bundle.ts:104-198`) are
  string-based — must run per language.

### 3.2 Registry / versioning language model
Today `(caseId, contentVersion)` is the PK; `latestPublishedVersion` picks
by `created_at DESC` (`registry.ts:61-73`). There is **no locale
dimension** — publishing `case@1.0-fr` would make French the "latest" for
everyone. Options:
- Add `locale` to the mysteries row + locale-aware latest-published lookup,
  cache keys, asset routes, and playthrough pinning (clean model, schema
  migration).
- Per-language case ids (`blackwood-inheritance-fr`) — zero code changes
  but forks catalog, grants, stats, and playthrough history.
Playthroughs pin `contentVersion` forever and `openingNarration` is
snapshotted onto the row (`db.ts:172,209`) — good: a run stays in its
starting language even after the user switches.

### 3.3 Studio
Local-only authoring tool editing raw `definition.json` with Zod
validation — no structured prose editing, no side-by-side source/target,
no change tracking. A real translation workflow (extraction, diff
tracking, per-language dirs) would need to be built; optional but valuable
once phase A starts.

### 3.4 Assets / audio / personas
- Covers + portraits have no baked-in text (verified visually); no audio in
  bundles; no voiced dialogue — all speech is text. **Add an authoring
  rule: never bake text into images** (sibling to the spoiler rule at
  `CASE_AUTHORING.md:1024-1028`).
- Ambient audio selection is keyed by theme/weather ids — fine; the pack
  *descriptions* in `lib/ambience.ts` are UI strings (§1.1).
- `content/personas/` is an empty catalog (README only) — build it with
  per-locale strings from day one.

## 4. Data model & API additions

- `users.locale` (account preference; drives playthrough creation and web
  default). Consider `Accept-Language` as first-visit hint only.
- `PlaythroughState.locale` — set at start, immutable for the run (keeps a
  case self-consistent; matches contentVersion pinning).
- API errors: codes, not prose (§1.6). Keep human `message` for debugging
  but clients must key on `error`.
- `GET /v1/cases` and case detail: locale negotiation (phase A) — serve the
  requested locale's version, fall back to English.
- Engine progress labels → structured data formatted client-side, or
  locale-aware engine (§1.6).

## 5. Phased plan

**Phase 0 — foundations (no user-visible change)**
API error codes; structured progress labels; pin machine tokens in
director/classifier prompts; authoring rule "no text in images";
translatable-field manifest doc; `Intl.*` helpers.

**Phase 1 — UI localization (site UX in N languages)**
`next-intl` + extraction (~300–400 strings), `[lang]` routing + metadata/
sitemap/manifest/OG plumbing, font subsets (+ Cinzel fallback, CJK stack),
locale switcher (nav) + `users.locale`, locale-keyed sessionStorage caches.

**Phase 2 — performed-language gameplay (option B)**
`PlaythroughState.locale` + prompt injection at the 3 call sites; fix the
English-dependent parsers (accusation scoring first, then presence
validator, then input classifiers — per-language packs or LLM-moved
detection); localize engine-generated stage-direction/hint strings
(`turn-pipeline.ts`, `boundaries.ts`, `accuse-gate.ts`, `effects.ts`,
`player-impact.ts`, `progress.ts`); token-budget recheck per script;
heavy QA per language (this is the cost `STEAM.md` warns about).

**Phase 3 — translated content per proven market (option A)**
Registry locale dimension; translation pipeline (extraction from the
manifest, CAT workflow or Studio tooling); per-language `lintBundle`;
voice/dialect re-authoring guide; per-language case QA incl. accusation
rubric alignment.

## 6. Top risks

1. **Accusation scoring** (`accusation.ts`) — English substring/negation
   matching decides win/lose; silent wrong verdicts in other languages.
2. **Boundary regexes are a safety layer** (`boundaries.ts`) — non-English
   jailbreak/abuse slips the local net, leaving only the LLM director.
3. **Accuse-gate confirm/withdraw** — untranslated, a pending accusation
   soft-locks (can never be confirmed) in another language.
4. **Dialect/period voice** ("East End vowels", Victorian register) has no
   mechanical mapping — needs localization design, not translation.
5. **Prefix-cache split** — per-language static blocks multiply cache
   misses and bundle storage; key caches by language explicitly.
6. **CJK** — no font coverage anywhere (UI + OG images), 2–4× token
   density, and the most regex-rework per language. Treat CJK as its own
   milestone, not "just another locale".

## 7. Open questions

- Which launch languages? (Drives font subsets, QA budget, regex packs.
  Latin-script European languages are dramatically cheaper than CJK.)
- Is performed-language quality (phase 2/B) acceptable for marketing, or
  only as an unlisted experiment? A bad French narration is worse than no
  French.
- Do translated cases share progression/leaderboards/grants with the
  English original (registry model choice in §3.2)?
- Locale switch mid-run: force finish-in-starting-language (recommended)
  or allow switch (breaks caches, mixed-language character memory — note
  player input is stored verbatim into memory and replayed into prompts,
  `memory.ts:25`, `context-pack.ts:504-505`)?
